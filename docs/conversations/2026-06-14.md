> When I change the audio output device in settings I sometimes get the following error.
> error: process didn't exit successfully: `target\debug\tonesmithy.exe` (exit code: 0xc000041d)

### [17:52:27] Claude (Opus 4.8)
Diagnosed and fixed a device-switch crash on the M17 branch (the affected files are identical on `development`, so this is a pre-existing bug, not M17 work).

#### Diagnosis
The exit code is the tell. `0xC000041D` = `STATUS_FATAL_USER_CALLBACK_EXCEPTION` — what Windows raises when a panic unwinds out through a Win32 callback. For an eframe app that means a panic inside `eframe::App::update`, which winit invokes from the window procedure. `apply_device_change` runs inside `update`, so a panic there surfaces as exactly this code. (Notably *not* `0xC000001D`, which is what our audio-thread `process::abort()` produces — so this is a main-thread panic, and the audio callback's `catch_unwind` is irrelevant.)

Root cause: cpal 0.15.3's WASAPI backend is full of `unwrap`/`expect` on COM calls (`stream.rs:163` `join().unwrap()` in the stream's `Drop`; `device.rs` activation/enumeration `expect`s). Building a new stream — or dropping the old one — on a device that has become unavailable or is already in use makes one of those COM calls fail, and cpal panics instead of returning `Err`. "Sometimes" = only when the device is in such a state.

#### Fix (chose: guard the switch; stay on M17 branch)
- `synth-host/src/audio.rs`: wrapped `build_output_stream` + `play()` in `catch_unwind`, surfaced via a new `AudioError::BackendPanic`. A bad switch now fails gracefully with an on-screen error instead of aborting the process. (Debug builds; release still aborts under `panic = "abort"`.)
- `synth-app/src/main.rs`: fixed a latent bug — the replacement engine was sized from the OS *default* device's sample rate via `default_output_format()`, not the device actually being selected. Now uses a new `audio::output_format(name)` for the target device, falling back to the current stream's rate.
- Tidied the CPU-load divisor that truncated sample rates above 65535 Hz through an `as u16` cast.

Verified: `cargo fmt --all --check`, `cargo clippy --workspace --all-targets -D warnings`, and `synth-host` tests all pass. Flagged that under `panic = "abort"` (release) the guard is a no-op, so a complete fix for shipped builds would mean relaxing `panic = "abort"` or upgrading cpal — offered as a follow-up.

### [17:52:27] Claude (Opus 4.8)
Recommended relaxing `panic = "abort"` over upgrading cpal, then implemented it.

Checked the registry: latest cpal is 0.18.1 (we're on 0.15.3), but the panic site is unchanged — the WASAPI `Stream::drop` still does `self.thread.take().unwrap().join().unwrap()`, and the COM `expect`s are still there. So a 0.15 -> 0.18 jump is three breaking-API releases of migration *and wouldn't fix the bug*. Recommended against it.

Instead, dropped `panic = "abort"` from `[profile.release]`:
- It's the only option that makes the existing `catch_unwind` guard work in shipped builds (catch_unwind is a no-op under abort).
- The audio-thread abort guarantee (design-patterns §2.8) is preserved — it's enforced by the explicit `catch_unwind -> process::abort()` in the audio callback, independent of the global panic strategy.
- Trade-offs noted: marginally larger binary + unwind tables (negligible here); panics elsewhere now unwind rather than abort, but an uncaught panic in the egui update still crashes via the window proc anyway, so behaviour for unrelated bugs is essentially unchanged.

Added an inline comment in `Cargo.toml` and a paragraph in design-patterns §2.8 so the setting isn't reintroduced. Verified `cargo fmt --check`, `clippy -D warnings`, and a `--release` workspace check.
