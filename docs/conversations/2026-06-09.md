Diagnosed and fixed — it was a state-desync bug from M13, not M14.

**Root cause:** `AppSettings` is cloned twice — `AppShell` (in `synth-app`) owns
one copy and `ToneSmithyApp` (the UI) owns another. `apply_device_change` only
updated the shell's copy, but the Settings combo boxes derive their
`selected_text` and selected-row highlight from the UI's copy. So after a

the selection looked like a no-op — even though the real stream had switched.

**Fix:** added `set_audio_device` / `set_midi_port` setters on `ToneSmithyApp`
and called them from `apply_device_change` on success only (a failed device open
must not display as selected). This also keeps the first-run wizard's pickers in
sync, since both go through `apply_device_change`. `fmt` + `clippy -D warnings`
clean; app builds.

The connected port `V61` is the correct keys port (`MIDIIN2 (V61)` is the Alesis
editor/control port), so it isn't a wrong-sub-port case — the right port is open
but delivering no note data.

in the midir callback (`75c3ab1`) so we can see whether the V61 transmits to us
at all. Asked the user to run `cargo run -p synth-app` from a terminal and press
keys: log lines appearing ⇒ delivery works, dig downstream; no lines ⇒ the port
receives nothing, which is environmental — most likely another app or a stray

preset/mode issue, or a driver re-enumeration. Suggested killing stray processes,

### [14:24:04] **You**
> it still seems possible to get a note stuck on with no way to stop the note.

### [14:24:04] Claude (Opus 4.8)
Real gap: there was no user-reachable panic. The engine had an internal
`voices.all_notes_off()` (used only on arp toggle), but no `AllNotesOff` event,
no MIDI CC 120/123 handling, and no UI button — so a lost NoteOff (queue full, a
missed key-up, or a disconnect mid-hold) left a voice latched in sustain forever
with no recovery.

Added a first-class panic path:
- New `EngineEvent::AllNotesOff` → `VoiceManager::panic` (release all voices,
  clear sustain-deferred releases, drop the sustain latch) + `ArpEngine::clear`
  (empty the held set), so nothing can stay latched.
- MIDI CC 120 (All Sound Off) / CC 123 (All Notes Off) map to it, so a
  controller's or DAW's panic button works.
- A "Panic" button in the header bar.
- Preset load now emits `AllNotesOff` first (the `map_to_events` contract already
  expected a preceding all-notes-off), so a held note can't hang under the new
  patch's envelope.

Release semantics preserved (panic triggers each voice's release, not a hard cut)
so it stays click-free. Added tests for the engine silencing path and the CC
mappings. Full workspace `fmt` + `clippy -D warnings` clean; engine 175 tests and
host 18 tests pass; app builds.
