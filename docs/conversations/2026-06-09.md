#### What was built

**Settings persistence** (`synth-presets/src/settings.rs`)
`AppSettings` struct (audio device, MIDI port, last preset, first-run flag) serialised as RON in
`%APPDATA%/Tone Smithy/Tone Smithy/settings.ron`. `load_settings` / `save_settings` never hard-fail.

**Device enumeration** (`synth-host`)
`audio::list_output_devices()` and `audio::start_on_device(name)` — opens a named cpal device,
falling back to the OS default. `midi::list_ports()` and `MidiInputStream::start_on_port(name)` —
same pattern for midir ports.

**Live device switching** (`synth-app/main.rs`)
`AppShell` now checks `ui.take_pending_device_change()` after every frame. Audio switch: reads
current snapshot, rebuilds `Engine` at the new sample rate, replays all params, opens new stream.
MIDI switch: just reconnects midir to the named port.

**Settings tab** (`sections/settings.rs`, `Tab::Settings`)
Audio output dropdown, MIDI input dropdown, "Apply" triggers `DeviceChange`. Shows current status
and the full list of active MIDI Learn bindings with per-entry delete.

**MIDI Learn** (`app/midi_learn.rs`, `app/utils.rs`)
`Knob::param_key("filter_cutoff_hz")` enables the right-click "MIDI Learn" item.
Clicking deposits the key in egui memory; `tick_midi_learn()` reads it next frame, watches for the
first CC that moves more than 0.02, binds it, then routes that CC to a `ParameterChange` every frame.
`param_range_for_key` maps 18 preset key strings to their `(ParamId, min, max)` triples.
Bindings are saved into `Preset::midi_learn` on save and restored on load (both from the file dialog
and the browser).
Wired on: filter cutoff, filter resonance, master volume, pitch offset.

**First-run wizard** (`app/wizard.rs`)
Modal `egui::Window` shown until `settings.first_run_complete = true`. Same device dropdowns as
the Settings tab. "Get started" button saves settings and dismisses.

All CI checks pass: fmt, clippy, tests.

### [18:20:00] Claude (Sonnet 4.6)
Three bugs found and fixed:

**1. Routing fired every frame** — `tick_midi_learn` was sending `ParameterChange` unconditionally
each frame, even when the CC hadn't moved. With CC at 0.0 it forced every mapped parameter to
`range_start` (e.g. filter cutoff to 20 Hz), making the knob appear frozen. Fixed: now only fires
when the CC moves by more than 0.0001 per frame.

**2. UI local state not synced** — the engine received the new value but `self.filter_cutoff_hz`
etc. still held the old value, so the knob display disagreed and snapped back on the next drag.
Fixed: `sync_cc_to_ui(key, value)` updates the mirror field whenever routing fires.

**3. Only 18 knobs covered** — MIDI Learn was greyed out on most controls. Coverage expanded
to 32 knobs: all amp/env2 ADSR, osc level/detune/pan, sub level, LFO rates,
chorus/delay/reverb/drive parameters, arp BPM/gate/swing. Also corrected the ranges in
`param_range_for_key` (drive 1–20, chorus depth 0–15, delay time 0.001–2, reverb decay 0.1–30).

### [18:25:00] **You**
> in the plan it says midi learn should work on any knob, but there are still knobs that don't work with midi learn

### [18:45:00] Claude (Sonnet 4.6)
Architectural fix. Two root causes found:

1. **Wrong key format**: `snapshot_to_map` uses `"osc1_level"` (no underscore before the number)
   but the `.param_key()` calls used `"osc_1_level"`, so lookups silently missed.

2. **Whitelist too small / wrong architecture**: the `param_range_for_key` whitelist approach
   meant any knob without an explicit entry was permanently barred. The fix makes the Knob
   self-sufficient: when "MIDI Learn" is clicked, it deposits `(key, range_start, range_end)`
   into egui memory. The range is stored in `MidiLearnEntry` directly, so routing never needs
   a range lookup table.

Changes:
- `MidiLearnEntry` gains `range_start: f32` + `range_end: f32` (default 0.0 for old presets)
- Knob deposits three egui-memory items on learn click (key + range); `midi_learn_target` is now `Option<(String, f32, f32)>`
- `param_range_for_key` replaced by comprehensive `key_to_param_id` covering all ~120 params
- `sync_cc_to_ui` covers all fields so the knob display follows the CC
- Fixed osc key names (`osc1_level` etc.)
- 58 knobs now have `.param_key()` set (was 32): all EQ, drive asymmetry, chorus spread, delay lowcut, reverb predelay, env2 curves, unison controls, mod matrix amounts, FM operator level/ADSR/feedback, FM slot level/pan
