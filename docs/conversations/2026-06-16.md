> when I have the sequencer on and I play lots of notes quickly by dragging my mouse across the virtual keyboard, notes get stuck on and I can only turn them off with the panic button

### [11:45:00] Claude (Opus 4.8)
Diagnosed an orphaned-voice leak in the immediate-fire path. With arp/seq on, the first note into an empty held set is fired straight into a voice on key-down (to avoid one-block latency), but releases were funneled only through `process()`'s empty-state cleanup. A fast release→re-press (dragging) refills the held set before the next block, so `process()` never sees the empty state, the previously fired voice never gets a NoteOff, and it stays stuck until panic. Added `take_idle_note_off()` to both `SeqEngine` and `ArpEngine` (when the held set has just emptied but a gate is still open, close it and return the sounding note); the engine calls it right after each NoteOff so every immediate-fire is balanced by an eager release. Added an engine test that rapid-fires 12 press/release pairs with the sequencer on and asserts no active voices remain — confirmed it fails on the pre-fix handler and passes after. Full suite green (198 engine tests), clippy/fmt clean.
