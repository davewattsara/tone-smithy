### [18:31:42] Claude (Opus 4.7)
Wrote `docs/planning/04-tech-stack/unit-testing.md` — prescriptive unit-testing doc with sibling docs to follow for integration, snapshot, property, real-time, and benchmark testing as we approach the milestones that need them.

##### What it covers

- **Principles**: test behaviour not implementation, pragmatic coverage (no floor), tests are documentation, determinism mandatory.
- **Where tests live**: inline `#[cfg(test)] mod tests`, not the `tests/` directory.
- **What to test / not test**: DSP correctness, public APIs, invariants, edge cases, state machines — yes. Trivial getters, UI rendering, audio output, cross-thread, external library behaviour — no.
- **Naming**: verb-phrase describing behaviour, no `test_` prefix.
- **Structure**: Arrange / Act / Assert with blank-line separation.
- **Float comparison**: explicit tolerances with a table of typical ε by test category, helper `approx_eq` and `assert_approx_eq!` macro.
- **Determinism rules**: seed every PRNG, no time / IO / threads.
- **DSP-specific patterns**: worked examples for oscillator periodicity, zero-crossings, filter cutoff attenuation, ADSR transitions, voice-stealing policy, parameter clamping.
- **Per-crate strategy**: synth-engine carries the heaviest test surface; synth-ui only tests pure helpers; etc.
- **Doctest policy**: include for public functions with examples; failing doctests fail CI.

##### What it deliberately doesn't cover

Listed at top and bottom — integration, snapshot/audio, property, real-time safety, benchmarks, UI/visual — each deferred to a sibling doc near the milestone that needs it.
